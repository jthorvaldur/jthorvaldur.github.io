1. System Overview

corpus-pipeline is a reproducible document ingestion pipeline that takes any directory of raw files — emails, PDFs, Word documents, spreadsheets, images, calendar events, and more — and transforms them into a fully searchable, fact-extracted knowledge base with complete provenance tracking.

The system is purpose-built for processing legal document corpora. It handles the messy reality of real-world file collections: duplicates, attachments, garbled encodings, binary formats, and mixed file types. Every transformation is recorded in a manifest, so you can always trace a fact or a vector back to the exact byte in the original file.

What goes in
Any directory of raw files: .eml, .pdf, .docx, .xlsx, .ics, .vcf, .zip, images, code, and unknown binary formats.
What comes out
Markdown files with YAML provenance headers, dense+sparse vector embeddings in Qdrant, extracted facts in Postgres, and a knowledge graph of entities and relationships.
Core guarantee
Every output — every vector, every extracted fact — links back to its source file via a SHA256 content hash. The manifest is the unbroken chain of custody.
Target corpus: ~/legal_backup_20260228/ — 33,275 files, 4.4 GB. Validated against a 1,967-file subset across 6 real-data runs (5 passing, 1 bug found and fixed).

2. Architecture Diagram

The pipeline is a linear sequence of stages. Each stage reads manifest.jsonl, does its work, and writes new fields back. The manifest is the provenance chain.

Manifest as provenance chain: every stage reads and annotates the same manifest.jsonl file. Fields accumulate: scan adds the hash, identify adds metadata, dedup adds canonical flags, convert adds the md_path. You can always replay any stage.

3. Data Flow Detail

Each stage is a standalone Python module runnable with python -m pipeline.<stage>. All stages are idempotent: safe to re-run without duplicating work.

Stage 1 — scan pipeline/scan.py 11 tests

What it does

Walks the input directory tree, skips dotfiles and dotdirs, and fingerprints every file. Optionally uses the Rust acceleration layer (corpus_pipeline_rs) for parallel SHA256 hashing — 3.3× faster on benchmarks, expected 10–50× on multi-core GPU hosts with large corpora.

Input

A directory path (CLI argument or config/default.yaml pipeline.input_dir).

Output

Creates manifest.jsonl with one JSON line per discovered file.

Stage 2 — identify pipeline/identify.py 31 tests

What it does

Opens each file and extracts type-specific metadata using 15 specialized handlers. Skips entries that already have identified_at (idempotent). Applies a _str() helper to coerce all metadata values to strings before JSON serialization (fixes pypdf IndirectObject crash discovered in real-data run 2).

Input

manifest.jsonl from scan stage. Reads actual file bytes.

Output

Updates manifest.jsonl in place with type-specific metadata.

Manifest fields added

FieldTypeDescription
metadatadictType-specific extracted data (email_from, pdf_pages, ical_event_count, etc.)
identified_atISO tsTimestamp when identify ran
Graceful degradation: pypdf, python-docx, openpyxl are optional. Missing libs produce informative placeholder metadata instead of crashes. 
python -m pipeline.identify

Stage 3 — dedup pipeline/dedup.py 18 tests

What it does

Two-pass deduplication: Pass 1 groups by content_hash — identical byte-for-byte files. Alphabetically first copy is canonical. Pass 2 (email only) groups canonicals by semantic key email:{date[:16]}:{bare_from}:{normalized_subject}, handling Re:/Fwd: threading and timezone variations.

Input

manifest.jsonl from identify stage.

Output

Updates manifest.jsonl in place with dedup fields.

Manifest fields added

FieldTypeDescription
dedup_groupstrHash or semantic key
is_canonicalboolTrue = selected as the keeper
semantic_duplicateboolTrue only for email semantic dupes
deduped_atISO tsTimestamp when dedup ran
Non-canonical entries are never promoted by semantic dedup. Only canonical files from Pass 1 are eligible for semantic comparison. 
python -m pipeline.dedup

Stage 4 — convert pipeline/convert.py 39 tests

What it does

Converts each canonical file to Markdown with a YAML frontmatter header. Only processes is_canonical=True entries. Skips if sdata/md/{content_hash}.md already exists (unless --force). The YAML frontmatter carries complete provenance: source path, hash, type, dedup group, converted timestamp, and type-specific fields.

Input

manifest.jsonl from dedup stage. Reads actual file bytes.

Output

  • Creates sdata/md/{content_hash}.md for each canonical file
  • Updates manifest with md_path and converted_at

Manifest fields added

md_pathstrAbsolute path to generated .md file
converted_atISO tsTimestamp when convert ran

Stage 4b — concepts pipeline/concepts.py 59 tests NEW

What it does

Extracts entities and relationships from converted Markdown files using Tier 1 regex patterns (zero external dependencies). Produces concepts.jsonl — a content knowledge graph of nodes and edges. Tier 2 LLM extraction is stubbed and runs as --dry-run placeholder.

Tier 1 extractors (regex, always runs)

  • Dollar amounts: $X,XXX.XX → Amount nodes
  • Dates: ISO 8601 + natural-language (May 19, 2024) → Date nodes
  • Email addresses: → EmailAddress nodes
  • Case numbers: 2024 D 6724 / 2024-D-006724 → CaseNumber nodes
  • Phone numbers: US format → Phone nodes
  • People: from YAML frontmatter (email_from → Person/sender, pdf_author → Person/author)

Stage 5 — embed pipeline/embed.py 17 tests GPU required

What it does

Chunks converted Markdown files and encodes them into dense and sparse vectors, upserted into a Qdrant collection. Filters to EXTRACTABLE_TYPES with body ≥200 characters. Uses 512-word chunks with 64-word overlap. Refuses to connect to production Qdrant ports (6333/7333).

Models

  • Dense: BAAI/bge-base-en-v1.5 (768-dim, cosine)
  • Sparse: prithivida/Splade_PP_en_v1 (indices + values)

Input

manifest.jsonl (md_path) + sdata/md/*.md

Output

Qdrant collection test_corpus at localhost:9333

Qdrant payload per point

FieldDescription
source_idcontent_hash (links back to original file)
source_pathOriginal file path
source_typeemail, pdf, etc.
chunk_indexWhich chunk of this document
textFirst 500 chars of chunk
embedded_atISO timestamp
Safety guard: embed raises an error if qdrant_port is 6333 or 7333 (production). Only 9333 allowed. 
python -m pipeline.embed
python -m pipeline.embed --dry-run    # validate without connecting to Qdrant

4. Data Type Handlers (15 types)

Every file type has a handler in identify.py (metadata extraction) and convert.py (Markdown conversion). All 15 types have test fixtures.

8. Agent Architecture

Three types of agents collaborate to build and evolve this system, coordinating via git commits, METRICS.md, and VALIDATION.md:

joel.thorarinson
Architect / Human

Defines goals in INTENT.md, writes VALIDATION.md feedback, requests new features. Human judgement for priorities and production decisions. 10 commits (37% of total).

noreply@anthropic.com
Builder Agent (Claude)

Implements pipeline stages, writes tests, generates docs, fixes bugs found by the GPU evolve loop. Reads VALIDATION.md for feedback. 11 commits (41%).

evolve@gpu.corpus-pipeline
GPU Evolution Agent

Runs on a persistent GPU instance. Polls git for code changes, runs the pipeline against real data (~/legal_backup/), records metrics/errors to VALIDATION.md, pushes results back. 6 commits (22%).

Coordination Flow

Human: adds feedback to VALIDATION.md, commits
  ↓
Builder Agent: reads VALIDATION.md, implements fix, commits + pushes
  ↓
GPU Evolution Agent (evolve.py, --interval 300s):
  1. git pull --rebase  (picks up new code)
  2. python -m pipeline [scan, identify, dedup, convert]
  3. Record metrics → learnings.py → VALIDATION.md
  4. If failure: write error to VALIDATION.md, git push
  5. Builder Agent sees failure on next run, fixes it

Supporting Modules

learnings.py

File-backed learning store (learnings.jsonl) with optional Qdrant vectorization. Logs fixes, errors, metrics, decisions with category and timestamp. query() does semantic search (Qdrant) or keyword grep. Prevents repeating known mistakes across runs.

providers.py

Cost-aware LLM router. Routes to cheapest capable provider per task tier (free: vLLM/Ollama; cheap: OpenRouter; mid: Claude Haiku; expensive: Claude Sonnet). Logs all usage to costs.jsonl with token counts and estimated cost.

# 3. Dedup: mark canonical files

# Print metrics
